## General Questions

<Accordions>
<Accordion id="what-is-nucel" title="What is Nucel?">

Nucel is an open-source deployment platform that makes it easy to deploy Next.js, SvelteKit, and React Router applications to AWS. It provides a Vercel-like experience while giving you full control over your infrastructure.

Key features:
- Zero configuration deployments
- Uses your AWS account
- Open source (no vendor lock-in)
- Supports multiple frameworks
- Cost-effective at scale

</Accordion>

<Accordion id="how-much-cost" title="How much does Nucel cost?">

Nucel itself is **free and open source**. You only pay for the AWS resources you use:

- **Lambda**: ~$0.20 per 1M requests
- **CloudFront**: ~$0.085 per GB transferred
- **S3**: ~$0.023 per GB stored
- **CloudWatch**: ~$0.50 per GB logs

Most applications cost **less than $5/month** for moderate traffic.

</Accordion>

<Accordion id="vs-vercel" title="How is Nucel different from Vercel?">

| Aspect | Nucel | Vercel |
|--------|-------|--------|
| **Hosting** | Your AWS account | Vercel's infrastructure |
| **Cost** | AWS usage only | Subscription + usage |
| **Open Source** | Yes | No |
| **Team Size** | Unlimited | Plan limits |
| **Customization** | Full control | Platform constraints |
| **Analytics** | CloudWatch | Built-in |

Choose Nucel for full control and lower costs. Choose Vercel for a fully managed platform.

</Accordion>

<Accordion id="production-ready" title="Is Nucel production-ready?">

Yes! Nucel is designed for production workloads with:

- Auto-scaling Lambda functions
- Global CloudFront CDN
- Enterprise-grade AWS infrastructure
- Secure IAM roles and permissions
- CloudWatch monitoring and logs

Several production applications are already using Nucel successfully.

</Accordion>

## Deployment Questions

<Accordions>
<Accordion id="first-deployment" title="How do I deploy my first app?">

1. Install Nucel CLI:
```bash
npm install -g @nucel.cloud/cli
```

2. Configure AWS credentials:
```bash
aws configure
```

3. Deploy your app:
```bash
nucel deploy
```

That's it! Nucel will detect your framework and deploy automatically.

</Accordion>

<Accordion id="multiple-environments" title="How do I manage multiple environments?">

Use stacks to manage different environments:

```bash
# Deploy to development (default)
nucel deploy

# Deploy to staging
nucel deploy --stack staging

# Deploy to production
nucel deploy --stack production
```

Each stack maintains isolated AWS resources and configuration.

</Accordion>

<Accordion id="preview-deployments" title="How do preview deployments work?">

Use the `--preview` flag to see changes before deploying:

```bash
nucel deploy --preview
```

For branch deployments, use dynamic stack names:

```bash
nucel deploy --stack feature-${BRANCH_NAME}
```

This creates isolated environments for each branch.

</Accordion>

<Accordion id="rollback" title="How do I rollback a deployment?">

Nucel doesn't have built-in rollback, but you can:

1. **Redeploy previous commit**:
```bash
git checkout previous-commit
nucel deploy --stack production
```

2. **Use different stack**:
```bash
nucel deploy --stack production-stable
```

3. **AWS Lambda versioning** (manual):
- Use Lambda aliases for blue-green deployments
- Configure CloudFront to point to specific versions

</Accordion>

<Accordion id="cicd-setup" title="How do I set up CI/CD?">

**GitHub Actions:**
```yaml
- name: Deploy
  run: |
    npm install -g @nucel.cloud/cli
    nucel deploy --stack production --yes
  env:
    AWS_ACCESS_KEY_ID: ${{ secrets.AWS_KEY }}
    AWS_SECRET_ACCESS_KEY: ${{ secrets.AWS_SECRET }}
```

**GitLab CI:**
```yaml
deploy:
  script:
    - npm install -g @nucel.cloud/cli
    - nucel deploy --stack production --yes
```

See [CI/CD Guide](/docs/deployment/cicd) for complete examples.

</Accordion>
</Accordions>

## Configuration Questions

<Accordions>
<Accordion id="env-variables" title="How do I set environment variables?">

Nucel loads environment variables from:

1. **`.env` files** (automatic):
```bash
.env                # Always loaded
.env.local          # Local overrides
.env.production     # Production only
```

2. **`nucel.config.ts`**:
```ts
export default {
  env: {
    API_URL: 'https://api.example.com',
    SECRET: process.env.SECRET
  }
}
```

3. **CLI flags**:
```bash
nucel env set API_KEY abc123 --stack production
```

</Accordion>

<Accordion id="custom-domain" title="How do I use a custom domain?">

1. Create ACM certificate in **us-east-1**:
```bash
aws acm request-certificate \
  --domain-name app.example.com \
  --validation-method DNS
```

2. Add to `nucel.config.ts`:
```ts
export default {
  domain: 'app.example.com',
  certificateArn: 'arn:aws:acm:us-east-1:...'
}
```

3. Deploy:
```bash
nucel deploy --stack production
```

4. Update DNS to point to CloudFront distribution.

</Accordion>

<Accordion id="build-command" title="How do I customize the build command?">

Override the build command in `nucel.config.ts`:

```ts
export default {
  buildCommand: 'npm run build:custom',
  // or for different stacks
  stacks: {
    production: {
      buildCommand: 'npm run build:prod'
    }
  }
}
```

</Accordion>

<Accordion id="aws-region" title="How do I change the AWS region?">

Set the region in multiple ways:

1. **Config file**:
```ts
export default {
  region: 'eu-west-1'
}
```

2. **Environment variable**:
```bash
AWS_DEFAULT_REGION=eu-west-1 nucel deploy
```

3. **CLI flag**:
```bash
nucel deploy --region eu-west-1
```

</Accordion>

<Accordion id="function-config" title="How do I configure Lambda settings?">

Customize Lambda settings in `nucel.config.ts`:

```ts
export default {
  // Memory in MB (128-10240)
  memorySize: 2048,
  
  // Timeout in seconds (1-900)
  timeout: 30,
  
  // Runtime version
  runtime: 'nodejs20.x',
  
  // Environment-specific
  stacks: {
    production: {
      memorySize: 3008,
      timeout: 60
    }
  }
}
```

</Accordion>
</Accordions>

## Troubleshooting Questions

<Accordions>
<Accordion id="aws-credentials-error" title="AWS credentials not found error">

**Solutions:**

1. **Configure AWS CLI**:
```bash
aws configure
```

2. **Set environment variables**:
```bash
export AWS_ACCESS_KEY_ID=your_key
export AWS_SECRET_ACCESS_KEY=your_secret
export AWS_DEFAULT_REGION=us-east-1
```

3. **Use AWS Vault**:
```bash
aws-vault exec personal -- nucel deploy
```

4. **Check credentials**:
```bash
aws sts get-caller-identity
```

</Accordion>

<Accordion id="build-failed" title="Build failed error">

**Common causes:**

1. **Missing dependencies**:
```bash
npm install
```

2. **Wrong Node version**:
```bash
nvm use 20
```

3. **Build script error**:
```bash
# Test locally first
npm run build
```

4. **Memory issues**:
```ts
// nucel.config.ts
export default {
  buildMemory: 4096 // Increase build memory
}
```

</Accordion>

<Accordion id="deployment-timeout" title="Deployment timeout">

**Solutions:**

1. **Increase timeout**:
```ts
export default {
  timeout: 60 // seconds
}
```

2. **Optimize function size**:
- Remove unnecessary dependencies
- Use tree-shaking
- Enable minification

3. **Check CloudWatch logs**:
```bash
nucel logs --stack production
```

</Accordion>

<Accordion id="cold-starts" title="How do I reduce cold starts?">

**Strategies:**

1. **Enable warmer**:
```ts
export default {
  warmerEnabled: true,
  warmerSchedule: 'rate(5 minutes)'
}
```

2. **Increase memory** (faster CPU):
```ts
export default {
  memorySize: 2048
}
```

3. **Optimize bundle size**:
- Remove unused dependencies
- Use dynamic imports
- Minimize initial payload

</Accordion>

<Accordion id="monitoring" title="How do I monitor my application?">

**Built-in monitoring:**

1. **CloudWatch Logs**:
```bash
nucel logs --tail --stack production
```

2. **CloudWatch Metrics**:
- Lambda invocations
- Error rates
- Duration
- Throttles

3. **Custom monitoring**:
```ts
// Add custom metrics
console.log(JSON.stringify({
  metric: 'custom.metric',
  value: 123
}));
```

4. **Third-party** (Datadog, New Relic):
- Add as Lambda layers
- Configure in environment variables

</Accordion>
</Accordions>

## Advanced Questions

<Accordions>
<Accordion id="vpc-config" title="Can I use a VPC?">

Yes, configure VPC in `nucel.config.ts`:

```ts
export default {
  vpc: {
    securityGroupIds: ['sg-xxx'],
    subnetIds: ['subnet-xxx', 'subnet-yyy']
  }
}
```

<Callout type="warning">
VPC configuration increases cold starts and requires NAT Gateway for internet access.
</Callout>

</Accordion>

<Accordion id="database-connection" title="How do I connect to a database?">

**Best practices:**

1. **Use connection pooling**:
```ts
import { Pool } from 'pg';

const pool = new Pool({
  connectionString: process.env.DATABASE_URL,
  max: 1 // Lambda constraint
});
```

2. **RDS Proxy** (recommended):
- Handles connection pooling
- Reduces cold starts
- Better for serverless

3. **Serverless databases**:
- Aurora Serverless
- DynamoDB
- PlanetScale
- Neon

</Accordion>

<Accordion id="websockets" title="Does Nucel support WebSockets?">

Not directly. Lambda doesn't support persistent connections. Alternatives:

1. **AWS API Gateway WebSockets**
2. **Pusher/Ably** (third-party)
3. **Server-Sent Events** (one-way)
4. **Long polling** (fallback)

For real-time needs, consider a hybrid approach with Nucel for main app and separate WebSocket service.

</Accordion>

<Accordion id="file-uploads" title="How do I handle file uploads?">

**Recommended approach:**

1. **Direct S3 upload**:
```ts
// Generate presigned URL
const url = await s3.getSignedUrl('putObject', {
  Bucket: 'my-bucket',
  Key: 'file.jpg',
  Expires: 300
});

// Client uploads directly to S3
```

2. **Lambda has 6MB payload limit** for direct uploads

3. **Use multipart uploads** for large files

</Accordion>
